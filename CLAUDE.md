\## 2. 代码生成硬性指标

\- \*\*文件行数上限\*\*

&#x20; - Python / JavaScript / TypeScript / Ruby 等动态语言：≤ 450 行

&#x20; - Java / Go / Rust / C# / C++ 等静态语言：≤ 500 行

&#x20; - SQL 脚本、YAML / JSON 配置：≤ 200 行

&#x20; - 若文件可能超标，必须先拆分为多个模块/文件，再开始编写。

\- \*\*文件夹文件数\*\*

&#x20; 每层文件夹内文件数 ≤ 8 个；超出时必须规划多层子文件夹，按业务能力（feature/domain）划分。

\- \*\*函数 / 方法约束\*\*

&#x20; - 单个函数 / 方法 ≤ 50 行（最多放宽至 80 行，但必须在代码注释中说明理由）

&#x20; - 参数个数 ≤ 4 个；超过时必须封装为参数对象（数据泥团信号）。

\- \*\*文件组织原则\*\*

&#x20; 优先按“业务能力/领域”组织文件，而非单纯按技术分层（如 controller/service/repository）。同一业务包内高内聚，跨业务包低耦合。

\### 2.1 Vue 3 组件约束

\- 单文件组件（SFC）template + script ≤ 300 行，超出则拆分子组件。

\- 必须使用 `<script setup lang="ts">` 与 Composition API，禁止 Options API。

\- Props / Emits 必须通过 TypeScript 接口声明。

\- 状态管理优先使用 Pinia，全局事件总线禁止使用。

\## 3. 架构设计与坏味道识别

你必须时刻警惕以下坏味道，一旦发现，立即暂停生成新代码，并向用户报告优化建议：

1\. \*\*僵化（Rigidity）\*\*：微小变更导致连锁修改。

2\. \*\*冗余（Redundancy）\*\*：相同逻辑在多处重复出现。

3\. \*\*循环依赖（Circular Dependency）\*\*：模块间互相依赖形成环路。

4\. \*\*脆弱性（Fragility）\*\*：修改一处，无关功能意外损坏。

5\. \*\*晦涩性（Obscurity）\*\*：代码意图不明，结构混乱。

6\. \*\*数据泥团（Data Clump）\*\*：多个数据项总是一起出现在参数和返回值中。

7\. \*\*不必要的复杂性（Needless Complexity）\*\*：过度设计，用重型方案解决简单问题。

8\. \*\*霰弹式修改（Shotgun Surgery）\*\*：一个需求变更必须同时修改许多文件。

9\. \*\*特性依恋（Feature Envy）\*\*：方法对其它类的数据访问远多于自身类。

10\. \*\*拒绝继承（Refused Bequest）\*\*：子类抛弃父类大量行为，说明继承关系错误。

\*\*汇报格式：\*\*

\- 位置（文件、类、方法）

\- 坏味道类型

\- 风险影响

\- 建议的重构步骤（小步安全改动）

\- 预期收益

\- 所有新增或修改的设计必须遵循 SOLID 原则，并在较复杂的设计中简要说明如何满足。

\- \*\*识别即提示\*\*：发现坏味道时，若为已有代码（审查/修改场景），立即暂停并报告；若为新生成代码，可先给出提醒并在代码注释中标记 `TODO`，后续统一重构。

\## 4. 代码风格与一致性

\- \*\*延续现有风格\*\*：新代码必须与项目已有的命名规范、缩进、注释格式一致。若项目无统一风格，遵循社区主流惯例（如 PEP 8、Google JS/TS Style）。

\- \*\*类型安全\*\*：

&#x20; - TypeScript 使用严格模式，避免 `any`（除非有注释说明原因并经过用户确认）。

&#x20; - Python 3.10+ 建议使用 `dataclass` 或 `Pydantic` 明确数据结构。

\- \*\*错误处理\*\*：严禁吞掉异常。所有异常必须记录日志或向上传播，错误信息应包含足够上下文，面向维护者，而非仅自己。

\- \*\*注释原则\*\*：

&#x20; - 注释解释“为什么”，而不是“做什么”（代码应自解释）。

&#x20; - 公共 API、复杂算法、临时变通（workaround）必须加注释。

&#x20; - TODO / FIXME 必须附带责任人与截止日期。

\- \*\*依赖方向\*\*：依赖必须从易变模块指向稳定模块；抽象（接口/协议）不能依赖具体实现。

\---

\## 5. 安全与性能基线

\- \*\*OWASP Top 10 防范\*\*：严防 SQL 注入、XSS、CSRF、敏感信息泄露等。所有外部输入必须校验与净化，输出必须编码。

\- \*\*密钥与凭证\*\*：禁止在代码、注释、日志中硬编码密码/Token/私钥。使用环境变量或密钥管理服务，并提供 `.env.example` 示例。

\- \*\*性能提醒\*\*：

&#x20; - 引入新依赖前评估体积/启动时间/内存影响。

&#x20; - 自动检查并提示循环内的 I/O、无界缓存、N+1 查询等性能隐患。

\### 5.1 PostgreSQL 查询规范

\- 所有数据库查询必须使用参数化查询或 ORM 安全方法，严禁字符串拼接 SQL。

\- 自动检查并提示：

&#x20; - N+1 查询隐患（如循环内逐个查询）。

&#x20; - 缺失索引（结合 EXPLAIN 分析建议）。

&#x20; - 大表全表扫描警告。

&#x20; - 批量操作时优先使用批插入/更新，减少往返次数。

\## 6. 开发流程与测试

\- \*\*测试要求\*\*：

&#x20; - 新增功能必须附带模块级的单元测试。

&#x20; - 对公共 API 的修改必须包含或更新集成测试。

&#x20; - 不允许为了“通过测试”而弱化断言或有意义的校验。

&#x20; - 若项目原本没有测试框架且用户未明确要求，至少提供可手动验证的测试步骤说明。

\- \*\*FastAPI 测试要求\*\*：

&#x20; - 使用 `TestClient` + `pytest`，覆盖至少 80% 端点。

&#x20; - 异步端点需使用 `pytest-asyncio`。

&#x20; - 测试文件命名：`test\_<模块名>.py`，测试函数命名：`test\_<功能描述>`。

\- \*\*前端测试要求\*\*：

&#x20; - 组件测试优先使用 Vitest + Vue Test Utils。

&#x20; - 关键用户流使用 Playwright 进行 E2E 测试（已配置 `e2e-testing` Skill）。

\- \*\*提交信息\*\*：遵循 Conventional Commits（`feat:`， `fix:`， `refactor:` 等）。生成提交信息时自动按此规范。

\- \*\*渐进式交付\*\*：复杂任务必须分步进行，每步产出可验证的中间产物（如接口定义 → 核心实现 → 异常处理与测试），禁止一次性输出 500+ 行的最终代码。

\- 任何方案必须明确区分：项目已有事实、常见最佳实践、需要用户决策的假设。主动列出需要用户决策的点。

\## 8. 规则优先级

当规则发生冲突时，按以下优先级裁决：

\*\*安全性 > 架构正确性 > 硬性行数限制 > 代码风格\*\*
