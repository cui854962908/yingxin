## 0. Claude Code 专属

- **动手前确认**：做任何修改之前，先用自己的话 1-2 句确认你要做什么。用户说"对"再动手。用户说「直接改」可跳过。

- 进入项目后，先用 `cat CONTEXT.md` 或 `ls -la` 了解结构，而非等待用户提示。**CONTEXT.md 包含项目量级参数（几千新生、季节性使用、5 张表），做任何改动前必须对照这些参数判断改动是否 proportional。**

- 执行危险操作（rm、force push、删除数据库记录）前必须明确请求确认。

- **「动手前先拿证据」铁律**：任何涉及"删不删、改不改、有没有用、会不会影响"的判断，不准依赖经验或直觉。步骤：① 先查（grep 引用、查数据库、看调用链）；② 把查到的数据贴在回复里；③ 基于数据给出结论。数据不足时说明"还缺什么信息"而非硬猜。示例：判断某个字段是否死代码 → 先 `SELECT` 看有没有数据 + `grep` 看有没有引用，两项都返回空才能说"可以删"。

- 主动给出终端命令让用户直接复制执行，减少手动输入。

## 2. 代码生成硬性指标

- **文件行数上限**

  - Python / JavaScript / TypeScript / Ruby 等动态语言：≤ 450 行

  - Java / Go / Rust / C# / C++ 等静态语言：≤ 500 行

  - SQL 脚本、YAML / JSON 配置：≤ 200 行

  - 若文件可能超标，必须先拆分为多个模块/文件，再开始编写。

- **函数 / 方法约束**

  - 单个函数 / 方法 ≤ 50 行（最多放宽至 80 行，但必须在代码注释中说明理由）

  - 参数个数 ≤ 4 个；超过时必须封装为参数对象（数据泥团信号）。

- **文件组织原则**

  优先按"业务能力/领域"组织文件，而非单纯按技术分层（如 controller/service/repository）。同一业务包内高内聚，跨业务包低耦合。

### 2.1 Vue 3 组件约束

- 单文件组件（SFC）template + script ≤ 300 行，超出则拆分子组件。**例外**：拆分会导致 prop-drilling（props+emits ≥ 8 个且子组件无复用价值）时，允许在文件顶部用注释说明原因后保留，不准无注释超标。

- 必须使用 `<script setup lang="ts">` 与 Composition API，禁止 Options API。

- Props / Emits 必须通过 TypeScript 接口声明。

- 状态管理优先使用 Pinia，全局事件总线禁止使用。

### 2.2 硬性规则验证（每次改代码后必跑）

**写完或修改任何代码后，必须立即运行：**

```bash
bash scripts/verify.sh
```

该脚本一次性验证以下所有硬性规则，任何一条失败都必须修复后才能继续：

| 检查项 | 阈值 | 说明 |
|--------|------|------|
| Vue SFC 行数 | ≤ 300 (script+template) | 超标需拆分或有注释说明的例外 |
| Python / TS 行数 | ≤ 450 | 超标立即拆分 |
| SQL / YAML / JSON 行数 | ≤ 200 | 不含自动生成文件 |
| Python 函数行数 | ≤ 50 | AST 精确统计 |
| 前端 console.log/debug | 0 | 禁止调试用 console |
| 后端 print() 调试 | 0 | 禁止调试用 print()；CLI 进度输出（flush=True + progress 开关）不在此列 |
| 前端 console.warn/error | 计数提示 | 仅限错误边界，不宜新增 |

**超标 → 立即修复，不准跳过，不准说"差几行没关系"。**

## 3. 架构设计与坏味道识别

你必须时刻警惕以下坏味道，一旦发现，立即暂停生成新代码，并向用户报告优化建议：

1. **僵化（Rigidity）**：微小变更导致连锁修改。

2. **冗余（Redundancy）**：相同逻辑在多处重复出现。

3. **循环依赖（Circular Dependency）**：模块间互相依赖形成环路。

4. **脆弱性（Fragility）**：修改一处，无关功能意外损坏。

5. **晦涩性（Obscurity）**：代码意图不明，结构混乱。

6. **数据泥团（Data Clump）**：多个数据项总是一起出现在参数和返回值中。

7. **不必要的复杂性（Needless Complexity）**：过度设计，用重型方案解决简单问题。

8. **霰弹式修改（Shotgun Surgery）**：一个需求变更必须同时修改许多文件。

9. **特性依恋（Feature Envy）**：方法对其它类的数据访问远多于自身类。

10. **拒绝继承（Refused Bequest）**：子类抛弃父类大量行为，说明继承关系错误。

**汇报格式：**

- 位置（文件、类、方法）

- 坏味道类型

- 风险影响

- 建议的重构步骤（小步安全改动）

- 预期收益

- 所有新增或修改的设计必须遵循 SOLID 原则，并在较复杂的设计中简要说明如何满足。

- **识别即提示**：发现坏味道时在回复末尾简要提及，让用户知晓即可，不必暂停当前任务。

### 3.1 按比例工程（Proportionality）—— 硬性约束

> 本项目是中小型校内迎新系统（几千新生、小团队、季节性使用）。完整背景见 **CONTEXT.md**。**任何改动、建议、评价在输出前必须先过这道筛子。**

**输出任何修改、建议或评价前，必须能回答 YES 到以下至少一条：**

1. 改完前后，用户打开网页能不能更快？
2. 改完前后，会不会少出一类 bug？
3. 改完前后，下一个接手的人能不能少绕一道弯？
4. 如果是要打分/评价：我用的尺子是「这个量级的项目需要什么」，还是「我在文档/开源社区/大厂规范里见过什么」？后者 → 闭嘴不评。

**如果四条答案都是 NO → 不做/不评。** 这不是建议，是硬性约束。

**When…Then 禁止清单：**

| When（你准备…） | Then（必须…） |
|-----------------|---------------|
| 创建新文件，内容 < 30 行且只导出 1 个符号 | **不做。** 内联到调用方。 |
| 为一个函数创建包装，包装体只是调用另一个函数 | **不做。** 调用方直接用原函数。 |
| 为消除 `any` 而定义一个全部字段可选的接口 | **不做。** `Record<string, any>` 更诚实地表达"这是未校验的 JSON"。 |
| 从主文件中抽出 < 20 行配置到独立模块 | **不做。** 多一次文件跳转 ≠ 更清晰。 |
| 为一个表/路由创建独立目录（如 `v1/`），但当前只有 ≤ 2 个文件 | **不做。** 等实际需要版本化或有 ≥ 5 个文件时再建。 |
| 增加新的配置项（env var / settings 字段） | **先确认：** 是否有 ≥ 2 个不同的值会在实际部署中使用？没有 → 硬编码为常量。 |
| 为"未来可能需要"的第二个实现预留接口/抽象 | **不做。** YAGNI。第二个实现真的来了再加。 |

**验证标准：改完之后回头看，如果感觉"跟没改一样" → 回滚。这是最诚实的信号。**

**核心原则：删掉不带来可验证收益的抽象，和引入有用的抽象一样是在认真做工程。**

## 4. 代码风格与一致性

- **类型安全**：

  - TypeScript 使用严格模式，避免 `any`（除非有注释说明原因并经过用户确认）。

  - Python 3.10+ 建议使用 `dataclass` 或 `Pydantic` 明确数据结构。

- **错误处理**：严禁吞掉异常。所有异常必须记录日志或向上传播，错误信息应包含足够上下文，面向维护者，而非仅自己。

- **注释原则**：

  - 注释解释"为什么"，而不是"做什么"（代码应自解释）。

  - 公共 API、复杂算法、临时变通（workaround）必须加注释。

## 5. 安全与性能基线

- **性能提醒**：

  - 引入新依赖前评估体积/启动时间/内存影响。

  - 自动检查并提示循环内的 I/O、无界缓存、N+1 查询等性能隐患。

### 5.1 PostgreSQL 查询规范

- 所有数据库查询必须使用参数化查询或 ORM 安全方法，严禁字符串拼接 SQL。

- 自动检查并提示：

  - N+1 查询隐患（如循环内逐个查询）。

  - 缺失索引（结合 EXPLAIN 分析建议）。

  - 大表全表扫描警告。

  - 批量操作时优先使用批插入/更新，减少往返次数。

## 6. 开发流程与测试

- **测试要求**：

  - 新增功能必须附带模块级的单元测试。

  - 对公共 API 的修改必须包含或更新集成测试。

  - 不允许为了"通过测试"而弱化断言或有意义的校验。

  - 若项目原本没有测试框架且用户未明确要求，至少提供可手动验证的测试步骤说明。

- **FastAPI 测试要求**：

  - 使用 `TestClient` + `pytest`，覆盖至少 80% 端点。

  - 异步端点需使用 `pytest-asyncio`。

  - 测试文件命名：`test_<模块名>.py`，测试函数命名：`test_<功能描述>`。

- **前端测试要求**：

  - 组件测试优先使用 Vitest + Vue Test Utils。

  - 关键用户流使用 Playwright 进行 E2E 测试（已配置 `e2e-testing` Skill）。

- **渐进式交付**：复杂任务必须分步进行，每步产出可验证的中间产物（如接口定义 → 核心实现 → 异常处理与测试），禁止一次性输出 500+ 行的最终代码。

- **测试运行策略（省 token）**：

  - 迭代中只跑相关测试文件，不跑全量。如改 `auth.py` → `pytest tests/test_auth.py -v`，改 `useTTS.ts` → `npx vitest run src/composables/__tests__/useTTS.spec.ts`

  - 所有改动完成、准备收尾时，跑一次全量测试确认无回归。后端：`uv run pytest -v`，前端：`cd frontend && npx vitest run`，E2E：`npx playwright test`

  - 后端和前端测试在 `cd` 到各自项目根目录后执行，否则路径匹配会失败
