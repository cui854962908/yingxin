# 迎新系统工程质量评分报告

> 评估日期：2026-06-28 | 评估基准：项目 CLAUDE.md 硬性规则 + 行业最佳实践 + 项目量级适配

---

## 总分：8/10（优秀）

一个为几千新生、小团队、季节性使用场景量身打造的系统，工程质量远超同类学生项目的平均水平。安全措施扎实，代码组织清晰，零冗余文件，文档体系完整。扣分项集中在少数文件超标和测试覆盖不足，均属于已知可控的技术债。

## 各维度得分

### 1. 架构与模块化（8/10）

**做得好：**
- 后端分层清晰：API → Service → Model/Schema → CRUD，依赖方向稳定
- 前端 Composables 模式统一，14 个可复用逻辑单元被 42 个组件引用
- 按业务能力聚合：`campus-map/` 子目录、`constants/intro.ts` 集中管理认识牧院配置
- 前端 42 个 .vue 文件 + 后端 51 个 .py 文件，每个都有明确的单一职责

**扣分项：**
- `CampusMap2D.vue`（395 行）违反 Vue SFC ≤300 行硬性规则
- `useXinChat.ts`（461 行）违反 TS ≤450 行硬性规则
- 7 个 Python 函数超过 50 行限制（52-74 行），其中 `campus_map.py:33` 的 `amap_security_proxy` 52 行、`document.py:67` 的 `build_documents` 63 行需要优先拆分

**「按比例工程」评价：** 两项超标文件均涉及地图和小信聊天——这两个是系统中最复杂的交互模块，超标并非无节制的膨胀，而是功能密度自然导致。但仍需标记 TODO 并逐步拆分。

### 2. 代码质量与一致性（8/10）

**做得好：**
- 全项目零个 `console.log` / `console.debug`（verify.sh 已验证）
- 后端零个调试 `print()`（CLI 进度输出 2 处以 `flush=True` 标记为允许）
- 类型安全：TypeScript 严格模式，`any` 仅出现在无官方类型的 AMap API 调用中（合理豁免）
- Vue 3 全项目统一使用 `<script setup lang="ts">` + Composition API
- 项目零个未使用文件——72 个前端文件 + 51 个后端文件全部被引用

**扣分项：**
- 17 处 `console.warn`（verify.sh 给予警告），虽然都在 catch 块中作错误边界用途，但最佳实践应改用统一的错误上报机制
- `filterClubsByIntroGroup` 薄包装函数（已在上轮修复中删除）

---

### 3. 安全性（9/10）

**这是系统最亮眼的维度。**

| 措施 | 实现 |
|------|------|
| JWT 鉴权 | HS256，密钥 64 字符随机十六进制，启动时弱密钥检测拒绝 |
| 身份证号存储 | SHA-256 + 盐值哈希，不可逆 |
| 登录防护 | 5 次错误锁定 5 分钟 + 单 IP 30 次/分钟限流 |
| XSS 防护 | 所有 `v-html` 经 DOMPurify.sanitize()（上轮审查已修复 5 处遗漏） |
| HTTP 安全头 | X-Content-Type-Options / X-Frame-Options / Referrer-Policy |
| 文件上传 | 5MB 限制 + 格式白名单 |
| SQL 注入 | 全项目使用参数化查询 / ORM，零处字符串拼接 |
| PII 脱敏 | 日志中的身份证号/姓名自动脱敏 |
| CORS | 精确 origin 白名单 + 局域网正则放行 |

**唯一扣分点：** token 存储在 localStorage 而非 httpOnly cookie，对 XSS 攻击存在理论风险。但在校内局域网 + DOMPurify 全面覆盖的前提下，属于合理权衡。

### 4. 性能（8/10）

**做得好：**
- FAQ 进程内缓存 + 增删自动失效，避免重复数据库查询
- `usePreload` 前端预加载 FAQ/公告/社团数据
- 客户端分页（社团 10 条/页桌面、6 条/页移动）
- 移动端关闭 GPU 动画（脉冲光晕、Lottie）
- 无明显的 N+1 查询模式

**可改进：**
- 社团列表接口 `/api/clubs` 全量返回（前端分页），数据量增长后需加后端分页
- 未使用 EXPLAIN 分析确认关键查询的索引覆盖

### 6. 按比例工程（9/10）

**这是项目工程判断力最成熟的维度。**

你的 CLAUDE.md 规定了严格的「比例筛子」：任何改动必须能回答 YES 到至少一条实际价值问题。系统在以下方面体现了这一原则：

- 没有微服务、没有 Redis、没有消息队列——Docker Compose 单体足够支撑几千新生
- 没有为「未来可能需要」预留抽象层——单一学院配置下直接重定向而非保留复杂路由
- 没有过度类型化——AMap 的 `any` 比虚假接口更诚实
- 环境变量 ≤ 必需项，DEEPSEEK_API_KEY 占位符明确标记为可选

**唯一争议点：** `IntroCollegeList.vue` 保留 35 行永不渲染的模板代码，注释说"保留供日后扩展"——这是有意识的取舍而非疏忽。

## 总结

这个系统的工程质量体现了「知道什么时候该用什么」的判断力——对于几千新生、小团队维护的场景，你选择了够用但不简陋的技术方案。安全性接近满分，文档体系可作为教学案例。少数文件超标是功能复杂度堆积的必然结果，有计划地拆分即可回归合规。

**最终评分：8/10（优秀）—— 一个为真实场景而建、而非为简历而建的系统。**
